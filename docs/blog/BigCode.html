              # 大きすぎるコード

              たくさんの機能が集まったプロダクト。
              たくさんの外部連携がある。
              たくさんの処理仕様もある。
              とても自分の頭では整理できない。どうすればいいのか。と頭を悩ませてしまった。

              # ググってみた

              https://qiita.com/baby-degu/items/f9a70cc0dcb51061c530 (opens new window)
              このページに則って考えてみた。
              コンテキストは割とわかっている。
              ファイル配置もわかっている。ファイル配置で大まかな依存先はわかっている自分の現状に気づいた。でも大まかな理解でいいのだろうか？という疑問が自分の中にあるというのが問題というふうに問題を詳細かできた。
              クラス図はわかっていない。Astahなどソースをクラス図に自動でできるツールを調べてみようと思った。 ビルドをして小さく試してみる。これもやってみよう。 小さなバグを直す代わりにリリースチケットを見よう。結局これが大事なんだ！ということに気づいた。
              他人ができなくていいことは、できなくていいのだ。

              # さらにググってみた

              ライブラリをラップするのが諸悪の根源なのには同意する。
              ライブラリを意識の外に外したいのに、ライブラリはアップデートを重ねてくるよね。。。 リファクタリングもうまく効かないか。。。。
              結局これも変わるものだけ考えるのが良さそう。

              # Googleのソフトウェアエンジニアリングを読んでみた

              良い知見はあまり得られなかった。。
              小さく試すことが大事で、変更量が大事なのだと思う。

              # 結局のところ

              自分はPDMでもないんだし、案件レベルのことが分かればいいんだ。
              リリースチケットを確認していくことにして最近の流れをキャッチアップしてみる。
